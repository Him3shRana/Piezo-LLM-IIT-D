# Piezo-LLM Simulation Pipeline — Implementation Spec

**Purpose of this document:** Full plan for the MD simulation → RDF analysis → model comparison → agentic orchestration pipeline for Piezoelectric Molecular Crystals (PMCs), covering both LAMMPS+MACE and ASE/MACE-OFF23 engines. This is meant to be handed back for actual implementation.

---

## 1. Overall Goal

Run MD simulations (NVT and NPT) on 47 PMC compounds, using every available MACE-family model (and sub-models: small/medium/large per family), across two engines (LAMMPS+MACE, ASE+MACE-OFF23). Compare model outputs to determine which model/size gives the most reliable simulation for PMCs — judged via RDF structural fidelity (relative comparison, no DFT/experimental ground truth available) and stability metrics. Eventually wrap all of this in an agentic AI layer (candidate: Qwen3 8B) that can be told "simulate compound X at condition Y" and it handles dispatch, tracking, and analysis automatically.

---

## 2. File Types — What Each One Is For

| File | Purpose | Notes |
|---|---|---|
| `.cif` | Starting crystal structure (source of truth for atomic composition + unit cell) | Input only, never modified |
| `.data` (LAMMPS) | Converted structure LAMMPS can read — Cartesian coords, expanded symmetry, numeric atom types | Generated from CIF via ASE/OVITO, once per compound |
| `type_map.json` | Per-compound numeric-type → element symbol mapping | Required for LAMMPS pipeline only; generated alongside `.data` |
| `.in` | LAMMPS input script (the actual run recipe: read_data, pair_style, fix, run, dump, write_restart) | Auto-generated per run from a template, saved (not discarded) inside the run's output folder |
| `.extxyz` | **Single standardized trajectory format** used for all downstream analysis (RDF, comparisons) regardless of which engine produced it | ASE writes natively; LAMMPS writes via native `dump ... extxyz` if `EXTRA-DUMP` package available, else convert from `.lammpstrj` post-run |
| `.pdb` + `.dcd` | Visualization only (VMD) — topology + trajectory pair | Generated from `.extxyz` post-run via MDAnalysis; never used for analysis, only viewing |
| `restart` file (LAMMPS: `.restart`, ASE: checkpoint `.npz`/pickle) | Exact physics state (positions, velocities, thermostat/barostat internal state) enabling correct continuation | **Single file per run, overwritten each checkpoint** — see Section 5 |
| `state.json` | Machine-readable current run status (single file, overwritten, not per-checkpoint) | See Section 5 |
| `progress.log` | Human-readable chronological append-only log of what happened during the run | Append-only (grows over time, this one *does* accumulate lines — it's a log, not a state file) |
| native LAMMPS thermo log | Auto-generated by LAMMPS (`-log thermo.log`) — temp/energy/pressure per step | Kept separate from the custom `progress.log` |
| RDF output (`.csv`) | Final g(r) vs r per element pair, per run | Produced by the single shared RDF script |
| model comparison table (`.csv`) | Cross-model, cross-compound summary (RDF deviation from CIF-static baseline, cross-model agreement, stability, wall-clock cost) | Built incrementally as runs complete |

**Important distinction:** `state.json` / restart file = current status only, always overwritten in place. `progress.log` = full history, always appended. They are generated from the same checkpoint event but behave differently on disk.

## 3. Directory Structure (per run)

```
runs/PMC-023_lammps_mace-medium_nvt_300K/
  config.yaml            # what was requested (intent)
  PMC-023.data           # structure input (or symlink to shared copy)
  type_map.json          # per-compound type mapping
  generated.in            # the exact LAMMPS script that ran
  traj.extxyz             # standardized trajectory (analysis)
  structure.pdb           # VMD topology (frame 0)
  traj.dcd                # VMD trajectory
  restart.restart          # SINGLE restart file, overwritten each checkpoint
  state.json               # SINGLE state file, overwritten each checkpoint
  progress.log              # append-only human log
  thermo.log                 # native LAMMPS thermo output
  rdf_output.csv               # RDF results for this run
  job.pbs / job.o### / job.e###   # PBS submission + cluster stdout/stderr
```

One folder per (compound × engine × model-family × model-size × ensemble × temperature[/pressure]) combination.

## 4. Model Registry — Handles Families + Sub-models

No external config file, no shared imports between scripts (per your requirement — each script stays fully self-contained, stdlib only). Each of the 4 core scripts (Section 6) has its **own hardcoded `MODELS` dict at the top** — new model = one line added directly in the relevant script(s), no separate file, no cross-script dependency.

```python
# Hardcoded at the top of each script that needs it
MODELS = {
    "mace": {
        "small":  {"path": "~/.cache/mace/mace_small.model",  "lammps_pt": "~/.cache/mace/mace_small_lammps.pt",  "cutoff": 5.0},
        "medium": {"path": "~/.cache/mace/mace_medium.model", "lammps_pt": "~/.cache/mace/mace_medium_lammps.pt", "cutoff": 6.0},
        "large":  {"path": "~/.cache/mace/mace_large.model",  "lammps_pt": None, "cutoff": 6.0},  # no LAMMPS .pt yet
    },
    "mace-off23": {
        "small":  {"path": "~/.cache/mace/maceoff23_small.model",  "lammps_pt": None, "cutoff": 5.0},
        "medium": {"path": "~/.cache/mace/maceoff23_medium.model", "lammps_pt": None, "cutoff": 5.0},
    },
    "polarmace": {
        "small":  {"path": "~/.cache/mace/polarmace_small.model", "lammps_pt": None, "cutoff": 6.0,
                   "notes": "LAMMPS unsupported — ACEsuit/mace#1409 (missing fermi_level/external_field buffers)"},
    },
}
```

CLI selects with two flags: `--model-family mace --model-size medium`. Script validates the combination exists and that the required file (`.model` for ASE, `.pt` for LAMMPS) is actually present before launching, failing loudly with a clear message if not (e.g. PolarMACE + LAMMPS → explicit `NotImplementedError` with the issue number, not a silent crash deep in a LAMMPS pair_style error).

## 5. State Tracking — Single File, Always Overwritten (Not Per-Checkpoint)

**Critical requirement from you:** no new `restart_1000.restart`, `restart_2000.restart`, etc. Only ONE restart file and ONE state.json per run, overwritten in place every checkpoint.

```
restart.restart   ← overwritten every checkpoint (LAMMPS write_restart always same filename)
state.json        ← overwritten every checkpoint
progress.log      ← appended every checkpoint (this one grows, by design)
```

**LAMMPS side:**
```
write_restart   restart.restart      # NOT restart_${step}.restart — same name every time
```

**state.json contents (always reflects only the current/latest status):**
```json
{
  "pmc_id": "PMC-023",
  "engine": "lammps",
  "model_family": "mace",
  "model_size": "medium",
  "ensemble": "nvt",
  "temperature_K": 300,
  "last_completed_step": 2000,
  "target_step": 2000,
  "status": "completed",
  "restart_file": "restart.restart",
  "trajectory_file": "traj.extxyz",
  "last_updated": "2026-07-11T15:10:22"
}
```

**Skip/resume/extend logic (fixes the bug discussed — status label alone is never trusted):**
```python
def get_run_action(state_path, new_target_step):
    """
    Decide whether to skip, resume, or start fresh, by comparing
    last_completed_step against the NEWLY requested target_step —
    never by trusting the old 'status' string alone.
    """
    if not os.path.exists(state_path):
        return "start_fresh"
    with open(state_path) as f:
        state = json.load(f)
    last_done = state["last_completed_step"]
    if last_done >= new_target_step:
        return "skip"
    elif last_done > 0:
        return "resume"
    return "start_fresh"
```

Every run of any script calls this first, before doing anything else.

## 6. Core Scripts (4 total — not per-model, not one monolith)

Split by the two axes that are *genuinely* different in logic (engine, ensemble) — model is just a parameter (Section 4), not a reason for a new script.

1. `run_nvt_lammps.py`
2. `run_npt_lammps.py`
3. `run_nvt_ase.py`
4. `run_npt_ase.py`

Each is fully self-contained (no shared `.py` imports across scripts — stdlib + ASE/LAMMPS-Python only). Each handles: CIF→data conversion (if not already cached), model registry lookup + validation, `.in` file generation (LAMMPS scripts) or direct ASE MD loop (ASE scripts), state.json / restart / log management (single-file, overwrite pattern from Section 5), and verbose console output (Section 8).

Model-specific exceptions (e.g. PolarMACE + LAMMPS unsupported) are handled as explicit, visible checks inside the relevant script — not hidden in a shared utility.

## 7. Steps, Not Picoseconds — Argument Convention

**All CLI arguments use integer MD steps, never time units (ps/fs/ns).** This avoids ambiguity from timestep-dependent conversion and matches your restart/checkpoint model (`last_completed_step` vs `target_step` — both step counts).

```
--target-step 10000        # NOT --duration-ps 10
--checkpoint-every 500     # steps between state.json/restart overwrites, NOT time
--dump-every 1000          # steps between trajectory frame writes
--timestep 0.001           # only used internally to convert steps->physical time for thermo display; never an input unit
```

Example CLI:
```bash
python run_nvt_lammps.py \
  --pmc PMC-023 \
  --model-family mace --model-size medium \
  --temperature 300 \
  --target-step 10000 \
  --checkpoint-every 500 \
  --dump-every 1000
```

## 8. Verbose Console Output — Required During Every Run

Every script prints, at minimum:
- On startup: compound ID, engine, model family/size + resolved file path, ensemble, temperature (and pressure if NPT), target step, output directory (full path)
- On resume: explicitly states "Resuming from step X → target Y using restart.restart" (or "Starting fresh" / "Already complete, skipping")
- During run: periodic step progress (e.g. every N steps or every checkpoint) — current step / target step, instantaneous temperature/energy from thermo, elapsed wall-clock
- On checkpoint: confirmation that restart.restart / state.json were overwritten (not appended as new files)
- On completion: final summary — total wall-clock, output files written, path to trajectory for RDF step

Example expected console output:

## 9. RDF / Plotting — Single Shared Script

`rdf_compare.py` — one script, engine-agnostic, reads only `.extxyz` regardless of origin.

```bash
python rdf_compare.py --traj runs/PMC-023_lammps_mace-medium_nvt_300K/traj.extxyz \
                       --pairs all \
                       --out runs/PMC-023_lammps_mace-medium_nvt_300K/rdf_output.csv
```

Responsibilities:
- Load trajectory frames (ASE `read(..., index=":")`, native extxyz support)
- Enumerate unique element pairs automatically from the frame's species
- Compute g(r) per pair with periodic minimum-image convention, using cell info embedded in extxyz
- Also compute the **static CIF baseline** (nearest-neighbor distances from the unrelaxed `.cif`, single frame, T=0) for comparison
- Output: CSV of r vs g(r) per pair, plus a plotted figure per pair (or grid)
- Also computable in "compare mode": given two+ `.extxyz`/CIF-baseline inputs, output a deviation score per pair (for the model-comparison table)

## 10. Visualization — Separate From Analysis

Small converter script, run post-simulation, `.extxyz` → `.pdb` (frame 0, topology) + `.dcd` (full trajectory, via MDAnalysis) for VMD viewing. Never used for RDF or any quantitative step — visualization only.

## 11. Model Comparison Methodology (No DFT/Experimental Reference Available)

Since there's no ground truth, "best model" is judged by:
1. **Drift from static CIF geometry** — RDF/nearest-neighbor distances of the equilibrated trajectory vs the raw CIF's T=0 geometry. Large drift = possible structural instability.
2. **Cross-model agreement** — for the same compound/ensemble/temperature, compare RDF shape across model families/sizes. Consistent peak positions across independent models = higher confidence; an outlier is flagged.
3. **Numerical stability** — energy conservation / no NaN or blow-up, from thermo log.
4. **Cost** — wall-clock time per N steps, to weigh small vs medium/large tradeoffs.

Output: one running `model_comparison.csv`, one row per (compound × model × ensemble × condition), columns: RDF-vs-CIF deviation, RDF-vs-other-models deviation, stability flag, wall-clock/1000 steps, status.

## 12. Agentic AI Layer (Candidate: Qwen3 8B)

Agent acts as an orchestrator calling a small, fixed toolset — it does not generate simulation code itself:

| Tool | Wraps |
|---|---|
| `resolve_compound(name_or_id)` | CIF/type_map lookup |
| `resolve_model(family, size, engine)` | Section 4 registry + validation |
| `check_run_state(...)` | Section 5 skip/resume/fresh logic |
| `generate_input(...)` | `.in` file generation (LAMMPS) or ASE MD setup |
| `submit_job(...)` | PBS `qsub`, returns job ID |
| `poll_job(job_id)` | `qstat` status check |
| `run_rdf(...)` | Calls `rdf_compare.py` |
| `update_comparison_table(...)` | Appends to `model_comparison.csv` |

8B-class models are workable for this narrow, well-scoped tool-calling loop (parse intent → call tools in sequence → report back), provided tool schemas are strict and narrow. Multi-step conditional sweeps (e.g. "rerun only the ones that disagree") may need more scaffolding/retry logic than single-job requests. Reliability depends more on tight tool design (Sections 4–9) than on model size — this is largely already satisfied by the structure above.

## 13. Code Style Requirement

Every script: clean, composed comments throughout — not just docstrings at the top, but explaining *why* at each non-obvious step (e.g. why restart filename never changes, why steps not picoseconds, why extxyz not traj/plain-xyz), so the script is self-explanatory to read top-to-bottom without external documentation.
